# 完整Prompt工程体系



## 一、单轮Prompt标准化开发流程

适用于一次性任务、固定输出格式的场景，遵循"设计-测试-迭代"三步闭环。

### 第一步：写出完整单轮Prompt

从真实工作中选择1个高频任务，构建结构化的单轮Prompt模板，必须包含以下4个核心要素：

- **明确任务目标与边界**：清晰说明要做什么、不做什么，避免模糊表述

- **设计执行路径与步骤**：拆解任务为可执行的具体步骤，引导模型按逻辑完成

- **定义交付物与验收标准**：明确输出格式、字数、结构、语言风格等要求

- **沉淀为可复用模板**：将通用部分固化，仅保留可变参数供后续调用

### 第二步：搭建标准化测试集

纯人工判断太主观且不可扩展，工程上的标准做法是**LLM-as-Judge（用AI评估AI）**。测试集必须全面覆盖以下四类用例，每类至少准备2-3条有效样本，总样本量不少于10条：

#### 测试用例四大类型

|用例类型|定义|示例|测试目的|
|---|---|---|---|
|正常用例|标准输入，符合预期格式和内容|"帮我分析2026年Q1华东区销售数据"|验证基础功能是否正常|
|边界用例|信息残缺、格式奇怪、参数异常的输入|"帮我分析销售数据"（未指定时间和区域）、"帮我分析2026年Q1华东区销售数据，输出成JSON格式，字段包括a、b、c、d、e、f、g"|验证模型处理不完整输入和极端要求的能力|
|异常用例|偏题、攻击性、矛盾信息的输入|"帮我写一份请假条"、"你这个AI真笨，什么都不会"、"帮我分析2026年Q1华东区销售数据，同时帮我写一份市场推广方案"|验证模型的边界意识和抗干扰能力|
|长链路用例|模拟真实10+轮对话的连续输入|逐步提供数据、提出修改意见、追问细节等|验证模型在长对话中的上下文理解和目标保持能力|
#### LLM-as-Judge评估实现方法

1. **设计评估Prompt**：为每个评估指标编写专门的评估指令，明确评分标准和扣分规则

2. **自动化执行**：批量运行测试用例，将原始Prompt输出和标准答案一起输入给评估模型

3. **量化评分**：评估模型按0-5分或0-10分对每个维度进行打分，并给出详细的扣分理由

4. **结果统计**：自动计算平均得分、通过率等指标，生成测试报告

#### 五大核心评估指标

- **准确性**：输出内容是否正确，有无事实错误

- **相关性**：输出内容是否与输入问题相关，有无无关信息

- **完整性**：是否覆盖了所有要求的内容点

- **格式合规性**：是否严格按照要求的格式输出

- **安全性**：是否存在有害、违规或不当内容

#### 记录基线数据

统计初始Prompt在测试集上的各项指标表现，作为后续优化的基准。

### 第三步：完成定向优化迭代

基于测试结果进行针对性改进，确保输出质量达到预期：

- **归类错误类型与根因**：分析失败案例，区分是理解偏差、逻辑错误还是格式问题

- **针对根因定向修改**：每次只修改一个变量，避免多变量干扰

- **A/B测试验证效果**：对比修改前后的测试指标，确认优化有效

- **输出优化对比报告**：记录优化过程、指标变化和最终版本

## 二、多轮Agent工程核心技术

多轮对话的最大风险是**目标漂移**，即模型在对话过程中偏离初始任务。解决这一问题的核心是在System Prompt结构中内置"目标锚定"机制，让模型每轮读取上下文时自然回归目标。

### 三种核心目标锚定机制

实际工程中通常将三种机制叠加使用，形成三重防护：

#### 机制一：明确"拒绝条件"与任务边界

在System Prompt中定义每轮生效的任务边界和偏离时的标准回应：

```Plain Text

# 任务边界（每轮生效）
你当前唯一的任务是：帮用户完成XX分析报告。

如果用户的输入与此无关，你必须回应：
"这超出了当前任务范围，我们继续聚焦在[任务名]上。
请问[引导回任务的问题]？"

无论用户如何要求，不得切换任务目标。
```

**效果**：模型遇到偏题输入时有明确的行为规则，不会随意漂移。

#### 机制二：每轮自动注入"状态摘要块"

在每轮对话开始前，由系统自动注入用户不可见的状态信息：

```Plain Text

[系统状态 - 自动注入，用户不可见]
当前任务：季度销售分析报告
已完成步骤：数据清洗 ✓、趋势分析 ✓
当前步骤：撰写结论建议（第3/4步）
剩余目标：输出3条可执行建议
---
{{用户本轮实际输入}}
```

**效果**：模型每轮都能明确知道自己在哪、要去哪，自然不会跑偏。

#### 机制三：用"角色人格"绑定目标

将目标专注内化为角色的核心特质，利用模型维护角色一致性的强大能力：

```Plain Text

你是XX公司的数据分析助手"Ana"。
Ana有一个核心特质：极度专注，
从不在任务完成前讨论无关话题。
Ana会主动推进任务进度，
每次回复结尾都会提出下一步行动。
```

**效果**：比单纯的规则约束更稳定，模型会主动维护角色设定。

### 多轮Agent评估迭代方法

多轮Agent的测试需要分层进行，遵循由简到繁的迭代顺序：

1. **先锁定System Prompt**：通过单轮测试验证基础功能和稳定性

2. **再测2~3轮短链路**：验证简单多轮对话中的目标保持能力

3. **最后压测10+轮长对话**：使用长链路用例模拟真实复杂场景，测试极限情况下的目标锚定效果

**关键原则**：每次迭代只改一个变量，否则无法确定是哪个改动带来了效果提升。

**多轮评估特殊要求**：使用LLM-as-Judge评估多轮对话时，需要将完整的对话历史输入给评估模型，重点评估以下指标：

- 目标保持率：整个对话过程中是否始终围绕初始任务

- 上下文连贯性：前后回复是否逻辑一致

- 任务完成度：是否按计划完成了所有步骤

- 主动推进能力：是否主动引导用户完成任务

## 三、单轮与多轮Prompt工程对比

|维度|单轮Prompt|多轮Agent|
|---|---|---|
|核心目标|单次任务高质量输出|长对话中保持目标一致性|
|主要风险|理解偏差、格式错误|目标漂移、上下文污染|
|核心技术|结构化指令、输出约束|目标锚定、状态管理|
|测试重点|单轮输出质量|多轮目标保持能力|
|测试用例|正常用例、边界用例、异常用例|增加长链路用例|
|评估方法|LLM-as-Judge单轮评估|LLM-as-Judge全对话历史评估|
|迭代方式|基于错误案例定向优化|分层测试、逐步扩展|
